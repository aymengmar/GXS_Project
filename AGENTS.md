# Project Overview

We are building **GXS Delivery**, an internal delivery management and organization app.

Domain placeholder:

```txt
gxsdelivery.app
```

GXS Delivery is used by:

1. **Drivers**
2. **Warehouse Staff / Lager**
3. **Admins**

The app is for Hamburg, Germany first, but must support multiple warehouses from day one.

GXS Delivery is **NOT** a packet scanning app.

The company already uses a separate external de-in/de-out scanning app for:

- packet scanning
- tracking IDs
- delivery status
- proof of delivery
- route execution

GXS Delivery is the management layer on top:

- driver onboarding
- document uploads and approvals
- External Driver ID assignment
- own-car / company-car driver management
- company-car invoice approvals
- own-car mileage visibility
- AI ZIP-zone assignment
- manual daily count summaries
- admin reports and analytics
- GDPR requests
- audit logs

Do not build packet scanning, QR scanning, barcode generation, or delivery proof features unless the user explicitly changes the product scope.

## 2. Driver Types

There are exactly two driver types.

### Own-Car Driver

Own-car driver:

- uses personal vehicle
- earns higher packet rate
- does not submit car invoices
- logs daily mileage only for admin visibility
- monthly earnings = delivered packets × own-car packet rate

Own-car driver fields:

```txt
plate number
make/model
insurance provider
daily start km
daily end km
```

Mileage is **not** used for reimbursement calculation in MVP.

---

### Company-Car Driver

Company-car driver:

- uses company-owned vehicle
- earns lower packet rate
- must submit vehicle-related invoices
- every invoice requires Admin approval
- monthly earnings = delivered packets × company-car packet rate
- monthly company cost = earnings + approved invoices

Invoice categories:

```txt
fuel
maintenance
highway_toll
car_wash
tyres
parking
other
```

Fuel invoice fields:

```txt
litres
price_per_litre
odometer
station_name
receipt_photo
amount
date
```

Maintenance invoice fields:

```txt
workshop_name
odometer
next_service_km
receipt_photo
amount
date
```

Other invoice fields:

```txt
supplier
description
route_description when needed
receipt_photo
amount
date
```

Own-car drivers must not be able to submit invoices.

---

## 3. External Driver ID

Every driver has an **External Driver ID** from the external scanning app.

Rules:

- assigned by Admin only
- never auto-generated by GXS Delivery
- unique across the entire system
- read-only for Driver and Warehouse Staff
- searchable by Admin
- included in all reports and exports
- never reused after driver leaves
- required before driver can become ACTIVE
- every assignment/change must create an audit log

This ID links GXS Delivery with the external scanning system.

## 4. Driver Account Lifecycle

Driver lifecycle:

```txt
PENDING
  Driver registered, documents not fully reviewed.

INCOMPLETE
  One or more documents rejected, driver must resubmit.

APPROVED
  All documents approved, waiting for Admin activation.

ACTIVE
  External Driver ID assigned, warehouse assigned, rate set, capacity set.

SUSPENDED
  Temporarily blocked by Admin.

INACTIVE
  Left company / archived.
```

Driver can receive ZIP assignments only when:

```txt
status = ACTIVE
external_driver_id exists
all required documents are approved
warehouse is assigned
packet rate is set
max daily packet capacity is set
home ZIP is geocoded
```

### Step 3 — Admin Review

Admin reviews every document.

If rejected:

- rejection reason is required
- driver status becomes INCOMPLETE
- driver receives email notification
- driver receives in-app notification
- driver must replace the document

When all required documents are approved:

```txt
driver status = APPROVED
```


Backend validates everything before status becomes ACTIVE.
